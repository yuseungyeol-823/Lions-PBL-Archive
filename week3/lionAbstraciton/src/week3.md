# Today I Learned (Week 3)

### 1. 오늘 배운 내용

- **추상화와 상속**: 공통된 속성(이름, 전공, 기수, 파트)을 부모 클래스로 분리하고, 역할별로 특화된 필드(학번, 직책)를 자식 클래스에서 정의하는 구조 학습.
- **인터페이스 활용**: '과제 제출 가능 여부'와 같은 정책(Policy)을 인터페이스로 추출하여 구현체에 따라 동작을 다르게 만드는 설계 실습.
- **다형성(Polymorphism)**: 상위 타입인 `Member`로 객체를 다루면서도, 실제 실행 시에는 자식 객체의 오버라이딩된 메서드가 호출되는 객체지향의 유연성 체감.

### 2. 핵심 정리 (내 언어로)

#### [1] 추상 클래스를 통한 정체성 확립

- **Member 추상 클래스**: 아기사자와 운영진 모두 '멋사 회원'이라는 공통점을 가집니다. 이를 `Member`라는 추상 클래스로 묶어 중복 코드를 제거하고, 반드시 구현해야 할 기능(`getRoleName`, `getExtraInfo`)을 강제하여 구조적 안정성을 높였습니다.
- **확장성**: 아기사자는 '학번'을, 운영진은 '직책'을 추가로 가지도록 설계하여 각 역할의 개성을 유지하면서도 부모의 기능을 상속받아 효율적으로 관리할 수 있습니다.

#### [2] 인터페이스를 통한 정책의 분리

- **Policy 인터페이스**: 과제 제출 여부를 판단하는 로직을 클래스 내부에 직접 쓰지 않고 별도의 `Policy` 객체에 위임했습니다.
- **유연한 설계**: 만약 "운영진도 특정 기간엔 과제를 낼 수 있다"는 규칙이 생겨도 `Main` 코드나 `Staff` 클래스를 수정할 필요 없이, 새로운 정책 클래스를 만들거나 기존 정책만 수정하면 되므로 변화에 매우 유연합니다.

#### [3] 조건문 없는 다형성 (No If-Else)

- **객체에게 맡기기**: 기존에는 역할을 확인하기 위해 `if (role.equals("LION"))` 같은 조건문을 써야 했습니다. 하지만 이번 미션에서는 `member.checkSubmissionStatus()` 한 줄만으로 각 객체가 자신의 정책(`LionPolicy`/`StaffPolicy`)에 따라 스스로 판단하게 만들었습니다.
- **결과**: `Main` 클래스는 객체가 정확히 무엇인지 몰라도 '회원이라면 당연히 할 수 있는 행동'만 요청하면 되기에 코드가 훨씬 간결해졌습니다.

### 4. 느낀 점

이번 미션의 핵심은 "묻지 말고 시키라(Tell, Don't Ask)"는 객체지향 원칙을 실현해보는 것이었습니다.

처음에는 단순히 학번과 직책을 구분하기 위해 `if`문을 쓰면 편할 것 같았지만, 상속과 인터페이스를 적용해 보니 왜 이런 복잡한 구조를 사용하는지 이해할 수 있었습니다. 조건문이 늘어날수록 코드는 수정하기 힘들어지지만, 다형성을 활용해 각 객체에게 판단을 맡기니 시스템이 훨씬 체계적으로 변했습니다.

특히 `Policy` 인터페이스를 통해 판단 로직을 외부로 꺼낸 설계는 매우 인상 깊었습니다. 기능을 직접 구현하는 것보다 '구조를 어떻게 짜느냐'가 소프트웨어의 유지보수성에 얼마나 큰 영향을 주는지 깨닫게 된 유익한 미션이었습니다.
